# Product

## Register

product

## Users

测试工程师、研发、质量负责人和平台管理员在日常测试资产维护、自动化调试、回归执行和问题追踪中使用本系统。用户通常需要快速定位用例、编辑步骤、复用运行环境、查看执行证据，并在真实本地浏览器或执行节点上完成调试。

## Product Purpose

自动化测试平台 2.0 用于统一管理用例、接口自动化、Web UI 自动化、缺陷、配置、AI 辅助生成和本地执行代理。成功的界面应帮助用户把复杂测试流程拆成可理解、可编辑、可验证的工作区，并减少在抽屉、表格、日志和运行工具之间来回切换的成本。

## Brand Personality

克制、专业、可信。界面应像面向长期工作的测试控制台：信息密度足够高，状态清晰，交互稳定，不用装饰性视觉抢走用户对步骤、变量、运行结果和错误证据的注意力。

## Anti-references

不做营销页式的大标题、装饰卡片和花哨动效；不把复杂编辑流程塞进越来越深的抽屉；不在页面里伪造浏览器。Web UI 录制和调试应由 Local Runner 打开真实浏览器完成，平台页面只承担控制台、步骤流、证据展示和编辑确认。

## Design Principles

- 任务优先：页面结构围绕创建、编辑、调试、保存和复盘，而不是围绕展示功能。
- 真实执行：Web UI 录制和运行以 Local Runner 的真实浏览器为准，页面不伪装运行环境。
- 渐进迁移：保留既有抽屉和列表入口兼容，复杂编辑逐步迁移到工作台式详情页。
- 状态可见：运行状态、步骤有效性、定位器、变量、截图和错误信息必须就近可见。
- 克制一致：沿用 Element Plus、现有 token 和平台组件，避免混入新的视觉语言。

## Accessibility & Inclusion

默认面向 WCAG AA：保证文本对比度、键盘可达、清晰焦点状态和可读的错误提示。动效只用于状态反馈，并尊重 reduced motion。复杂表单和步骤编辑需要明确标签、空状态和加载状态，避免只依赖颜色表达结果。
