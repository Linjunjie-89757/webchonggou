一、接口调试页面整体设计思路（对标 Apifox，适配你的自动化测试平台）
1、产品定位
延续 Postman 的调试操作手感，同时补齐 Swagger 结构化文档能力，做到调试区与文档区完全解耦，既满足接口快速调试，又解决导入 OpenAPI 文档后字段说明丢失的行业痛点。
面向人群：接口自动化编写人员，兼顾新手（可视化表单）和资深开发（JSON 代码模式）。
2、核心设计原则
（1）数据分层隔离（最核心）
把数据拆分为两块独立存储，互不覆盖，这是 Apifox 最关键的架构：
1）文档元数据（只读数据源）
导入 Swagger 时，解析出来的字段结构：字段名、数据类型、是否必填、描述、示例值，单独存入数据库，不会被用户调试时修改的 JSON 内容覆盖。
2）用户调试数据（可编辑）
用户自己填写的请求 JSON、请求头、参数，独立存储。
价值：重新编辑请求不会弄丢接口文档，文档和调试内容彻底分开。
（2）双视图模式，一键切换
1）代码视图：纯 JSON 文本编辑器，适合熟练人员手写报文；
2）表单视图：自动根据文档 Schema 自动渲染输入框，零代码填写，适合新手。
两个视图双向同步数据，改表单自动同步到 JSON，修改 JSON 也可以同步回表单。
（3）页面布局分层（贴合你现有双栏页面结构）
顶部：请求行（请求方式 + URL + 操作按钮：发送、保存、Curl）
主体一级 Tab 栏：请求头 → 请求体 → Query 参数 → 前置 / 后置脚本 → 断言 → 接口文档
请求体内部再嵌套二级切换：代码视图 / 表单视图
独立「接口文档」Tab：专门展示结构化字段表格，作为对照参考，还支持一键把示例值填充到请求报文
底部：响应结果面板
（4）交互体验规范（和你整套 UI 保持统一）
视觉：卡片分层、轻微阴影、圆角 8px，沿用全局色号；
必填项用红色标签醒目区分，不强制拦截发送，保持调试灵活性；
增加辅助能力：一键填充示例、一键清空报文；
后期扩展：JSON 编辑器光标悬浮字段，自动读取文档弹出 Tooltip 注释。
（5）和你现有系统的融合方案
该模块直接嵌入你当前「接口自动化→接口详情页」，左侧保留接口树形目录，右侧直接使用这套编辑器，不需要重构整体页面。
二、Swagger / OpenAPI 文档导入完整处理方案（后端 + 前端完整流程）
整体流程
用户上传 JSON/YAML 格式的 OpenAPI 文档 → 后端解析文档 → 拆分两类数据入库 → 前端渲染文档 + 自动生成表单结构。
步骤 1：后端解析文档，拆分两类数据
1）解析内容
读取 OpenAPI 3.0 / Swagger 2.0 文档，提取每个接口的元信息：
基础信息：接口路径、请求方式、接口名称、接口描述
路径参数（path）：字段名、类型、必填、说明、示例
查询参数（query）：字段名、类型、必填、说明、示例
请求体 JSON Schema：嵌套字段、object 对象、数组、字段类型、required 必填、description 描述、example 示例
响应体 Schema：返回字段结构
2）分两张表存储（关键拆分，避免数据互相覆盖）
表 A：接口基础 + 文档 Schema（永久不变，文档源数据）
用来渲染【接口文档】Tab + 自动生成表单视图
字段：
接口 ID
参数类型：path/query/body/response
字段完整路径（支持嵌套：user.username）
字段名称
数据类型 string/number/boolean/object/array
是否必填 required: true/false
字段描述 description
示例值 example
表 B：用户调试报文（可随时修改）
用来保存用户填写的真实请求内容
字段：
请求头 JSON 文本
Query 参数键值对
Body 原始 JSON 字符串
Cookie、超时、重试等配置
3）重新同步文档的冲突处理（Apifox 核心逻辑）
当用户再次导入新版本 Swagger：
提供两个选项：
覆盖模式：用新文档全量替换旧 Schema 结构；
合并模式：保留用户手动编辑的字段备注，只更新字段结构，不覆盖自定义描述。
步骤 2：前端业务逻辑落地
1）表单视图自动渲染
前端读取表 A 的 Body Schema 树形数据，自动循环生成表单输入框：
必填字段自动带红色 *；
每个字段附带问号图标，悬浮展示类型 + 描述 + 示例；
支持解析嵌套对象、数组多层级结构。
2）视图双向同步
表单输入内容 → 自动序列化生成 JSON 字符串，同步到代码编辑器；
手动修改 JSON → 自动反序列化，把值回写到表单输入框。
3）文档面板能力
读取全部 Schema 数据，渲染成表格：字段名、类型、必填、描述、示例；
增加按钮：一键填充所有示例值，自动把文档里的 example 批量写入请求体 JSON。
步骤 3：扩展能力（二期迭代）
JSON 编辑器（Monaco）光标悬浮在 key 上，向后端匹配字段文档，弹出悬浮提示；
发送请求前自动校验：扫描 JSON，高亮提醒缺失必填字段，仅做弱提醒，不阻止发送；
自动生成前后置提取脚本、断言模板；
自动识别 JSON 层级，支持多层嵌套树形文档展示。
三、落地开发清单（分两期）
一期（当前 demo 已实现）
页面 UI：请求行 + Tab 栏 + 双视图切换；
独立接口文档页面，表格展示字段；
一键填充示例、清空报文；
两套数据隔离：文档 Schema 和 用户 JSON 报文分离。
二期迭代
接入 OpenAPI 解析接口，自动生成 Schema 数据，替代前端硬编码数据；
Monaco 代码编辑器 + 字段悬浮 Tooltip；
支持嵌套对象、数组结构渲染；
报文与表单双向实时同步；
导入文档时支持覆盖 / 合并两种策略。
四、技术架构总结
存储架构：文档 Schema（静态只读）+ 用户报文（动态可编辑）双数据模型，解决 Apifox 文档不丢失的核心痛点；
交互架构：代码视图 + 表单视图双模式，覆盖新手与专业人员；
页面架构：主 Tab 区分调试区与文档区，调试写报文，文档做对照；
导入架构：解析 Swagger → 结构化入库 → 前端自动渲染表格与表单，彻底告别手动录入字段。
