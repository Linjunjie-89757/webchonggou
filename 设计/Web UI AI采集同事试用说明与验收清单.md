# Web UI AI 采集同事试用说明与验收清单

## 1. 文档目的

本文用于公司内部同事试用 Web UI 元素库的 AI 采集能力。

当前版本定位是“第一阶段可试用闭环”，目标是让同事能够：

- 启动本地 Local Runner。
- 在本机浏览器环境中打开被测页面。
- 对需要登录的页面完成手动登录后采集元素。
- 查看 AI 采集任务、候选元素、过滤明细和验证结果。
- 人工确认后保存元素到元素库。
- 从元素详情回跳来源采集任务。

本文同时明确当前未完成能力边界，避免试用时把一期能力误认为最终商业化形态。

## 2. 当前能力范围

一期已具备：

- Local Runner 本地采集入口。
- 前端检测 Runner 在线状态。
- Runner 打开目标页面。
- Runner 采集当前页面 DOM 候选、页面 URL、标题、截图等素材。
- 后端创建采集任务并记录任务状态。
- 后端规则清洗、去重、稳定性评分。
- AI 连接池参与候选增强，包括命名、分组、语义解释和噪声过滤建议。
- 采集工作台展示候选元素、全局截图、过滤明细、任务信息。
- 支持恢复部分被过滤候选。
- 支持当前候选重新验证。
- 支持保存候选到元素库。
- 元素详情可查看来源采集任务。
- 元素库顶部可查看最近采集任务和全部历史采集任务。
- 已完成、已失败、已取消、已降级任务可删除。
- Runner 健康检查可展示版本、Playwright、Chromium、会话状态、能力清单和限制值。
- 支持保存、清空和复用当前运行环境的登录态快照。
- 支持页面会话 TTL、会话释放、任务绑定和后台轮询拉取验证指令。

当前不应承诺：

- 不承诺 Runner 已产品化成正式安装包。
- 不承诺自动登录、验证码识别、扫码登录自动化。
- 不承诺所有复杂页面都能一次性生成完整元素库。
- 不承诺 iframe、shadow DOM、虚拟滚动、懒加载页面已经全部稳定覆盖。
- 不承诺服务端远程可视浏览器。
- 不承诺 WebSocket、MQ、Runner 心跳已经完整接入。

## 3. 使用前准备

### 3.1 启动前端和后端

确认后端已启动，并且前端能正常访问平台。

常用前端地址：

```text
http://localhost:4173
```

常用后端地址：

```text
http://localhost:8080/api
```

如果页面接口返回 401，通常是当前浏览器没有登录平台，先重新登录平台。

### 3.2 启动 Local Runner

在 PowerShell 里进入前端项目目录：

```powershell
cd D:\perfectproject\newautoweb\perfectprojectwebchonggou-main
npm.cmd run runner
```

启动成功后，终端应看到类似：

```text
Web UI Local Runner 0.1.0 listening on http://127.0.0.1:39118
```

Runner 健康检查地址：

```text
http://127.0.0.1:39118/health
```

如果浏览器或命令行访问该地址返回 200，说明 Runner 已在线。

### 3.3 Runner 检测不到怎么办

优先检查：

- 是否在正确目录执行了 `npm.cmd run runner`。
- 是否误在 `C:\Users\用户名` 目录下执行命令。
- `127.0.0.1:39118` 是否已被占用。
- 终端是否因为 Playwright 缺失、依赖缺失或脚本错误退出。
- 前端是否访问的是本机页面，而不是远程服务器页面。

可用命令检查端口：

```powershell
Get-NetTCPConnection -LocalPort 39118 -ErrorAction SilentlyContinue
```

如果没有输出，说明 Runner 没有监听该端口，需要重新启动。

如果 Playwright 浏览器内核缺失，可先执行：

```powershell
npx playwright install chromium
```

### 3.4 采集弹窗里的 Runner 诊断怎么看

AI 采集弹窗的本地 Runner 区域会展示：

- Runner 版本。
- Playwright 是否可用。
- Chromium 是否已安装。
- 当前页面地址。
- 页面会话是否有效。
- 登录态是否已保存。
- Runner 能力清单，例如有头浏览器、登录态快照、静态采集、真机验证、后台轮询、会话 TTL。
- 当前限制值，例如页面会话过期时间、单次最多验证的定位器数量、最多返回的截图证据数量。

如果 Runner 未连接、Playwright 不可用或 Chromium 缺失，弹窗会显示处理命令。优先按弹窗里的命令处理，再点击“检测”。

## 4. AI 采集使用流程

### 4.1 进入元素库

进入：

```text
Web UI 自动化 -> 元素库
```

建议先在左侧目录中选择合适范围：

- 空间
- 模块
- 页面对象
- 分组

如果还没有模块或页面对象，可以先创建模块、页面对象和分组。

### 4.2 打开 AI 采集入口

点击元素库顶部：

```text
AI 采集
```

弹出采集启动弹窗后，选择：

- 运行环境
- 所属模块
- 页面对象
- 分组策略
- AI 采集模型
- 采集范围

说明：

- AI 采集模型来自系统 AI 连接池配置。
- 如果选择默认 AI 建议分组，系统会根据候选元素语义推荐分组。
- 如果选择自选分组，保存时会优先使用指定分组。

### 4.3 打开本地页面

点击打开页面按钮后，Runner 会使用本机 Playwright 打开浏览器。

如果被测页面需要登录：

1. 在 Runner 打开的浏览器里手动登录。
2. 可以完成验证码、扫码、SSO、多因素认证等人工操作。
3. 登录后进入真正要采集的业务页面。
4. 回到平台点击继续采集。

注意：

- 采集的是 Runner 当前打开的真实页面。
- 如果 Runner 页面停留在登录页，采集结果会很少或直接失败。
- 如果页面跳到 404、500、空白页，也应重新打开正确页面后再采集。

### 4.4 开始采集

点击开始采集后，系统进入 AI 采集工作台。

工作台会展示：

- 任务状态。
- 阶段进度。
- 页面信息。
- 全局截图。
- 候选元素列表。
- 过滤统计。
- 过滤明细。
- AI 命名、分组和解释。
- 真机验证结果或降级提示。

### 4.5 任务阶段说明

常见阶段：

| 阶段 | 含义 |
| --- | --- |
| 上传快照 | Runner 正在把采集素材提交给后端 |
| 规则清洗 | 后端按规则过滤空节点、低价值节点、重复节点 |
| AI 分析 | AI 对候选做命名、分组、解释和噪声判断 |
| 真机验证 | Runner 对候选 locator 做本地真实页面验证 |
| 处理完成 | 后端生成最终候选列表 |

常见任务状态：

| 状态 | 含义 |
| --- | --- |
| 处理中 | 任务尚未完成 |
| 等待本地验证 | 后端已生成待验证 locator，等待 Runner 验证 |
| 已完成 | 任务正常完成 |
| 已降级 | 采集成功，但本地验证失败或超时，候选可能未验证 |
| 已失败 | 采集任务失败 |
| 已取消 | 用户主动取消任务 |

## 5. 候选元素处理

### 5.1 候选列表怎么看

候选列表重点看：

- 元素名称。
- 所属分组。
- 定位方式。
- 定位值。
- 稳定性评分。
- 来源。
- 验证状态。
- AI 解释。
- 维护建议。

定位方式建议优先级：

```text
Test ID -> Role -> Label -> Placeholder -> Text -> CSS -> XPath
```

说明：

- Test ID 最适合作为长期稳定定位。
- Role、Label 更贴近用户可感知语义。
- CSS 可用于业务页面已有稳定 class、id、name 的情况。
- XPath 作为兜底，不建议大量使用。

### 5.2 验证状态说明

| 状态 | 说明 |
| --- | --- |
| 已验证 | Runner 真机验证通过 |
| 未找到 | Runner 未找到该 locator |
| 多匹配 | locator 命中了多个元素，需要人工判断是否可用 |
| 未验证 | 当前候选没有经过 Runner 真机验证 |
| AI 建议未验证 | AI 补充或 AI 判断的候选，尚未通过强验证 |

保存前建议优先选择：

- 推荐保存。
- 已验证。
- 稳定性评分较高。
- 命名和分组符合业务语义。

### 5.3 过滤明细怎么用

过滤明细用于解释“为什么某些元素没进最终候选”。

常见过滤原因：

- 空 locator。
- 重复候选。
- 低稳定性。
- 无意义文本节点。
- 不可交互元素。

如果认为过滤错了，可以点击恢复。恢复后该候选会回到候选列表，建议再执行重新验证。

### 5.4 重新验证

适用场景：

- Runner 刚才离线。
- 页面还没加载完。
- 用户登录态刚恢复。
- 恢复了过滤明细中的候选。
- 候选显示未验证或 AI 建议未验证。

重新验证前请确认：

- Runner 正在运行。
- Runner 打开的页面还是目标业务页面。
- 页面没有跳回登录页。
- 弹窗、分页、筛选等页面状态符合当前要验证的元素。

## 6. 保存元素

保存前建议做人工检查：

- 元素名称是否清楚。
- 分组是否正确。
- locator 是否稳定。
- 是否重复保存已有元素。
- 验证状态是否可信。

保存后：

- 元素会进入元素库。
- 元素详情会记录采集来源。
- 可从元素详情点击查看来源采集任务。
- 保存完成后页面会尽量定位到对应页面对象或分组。

## 7. 采集任务管理

元素库顶部有：

```text
采集任务
```

这里包含：

- 最近采集任务。
- 查看全部采集任务。
- 打开历史任务。
- 按任务状态筛选。
- 按页面标题、URL、页面对象或任务 ID 搜索。
- 删除终态任务。

删除规则：

- 处理中任务不能直接删除。
- 需要先取消任务，再删除。
- 已完成、已失败、已取消、已降级任务可以删除。

说明：

- 删除采集任务不会自动删除已经保存到元素库的元素。
- 删除后无法从任务列表再次打开该历史任务。
- 最近任务只是快捷入口，移除最近任务不等于删除后端历史任务。

## 8. 常见问题

### 8.1 Runner 检测不到

原因通常是：

- Runner 没启动。
- 启动目录不对。
- 端口不是 39118。
- 脚本启动后退出。
- 浏览器跨域被 Runner 拦截。

处理方式：

```powershell
cd D:\perfectproject\newautoweb\perfectprojectwebchonggou-main
npm.cmd run runner
```

然后访问：

```text
http://127.0.0.1:39118/health
```

### 8.2 采集到的元素很少

优先检查：

- 当前 Runner 页面是否真的进入业务页面。
- 是否还停留在登录页。
- 页面是否为空白页。
- 业务内容是否需要滚动或点击后才出现。
- 是否选择了过窄的采集范围，例如只采集按钮。

### 8.3 真机验证全部未找到

常见原因：

- Runner 页面已经跳转。
- 用户手动刷新到了不同页面。
- 页面仍在登录页。
- 页面 DOM 发生变化。
- 候选 locator 来源于旧页面。

建议重新进入目标页面后重新采集。

### 8.4 AI 命名不准确

AI 命名是辅助建议，不是最终结论。

保存前可以手动改：

- 元素名称。
- 所属分组。
- 描述。
- 是否保存。

### 8.5 候选显示未验证能不能保存

可以保存，但不建议批量保存大量未验证候选。

未验证候选适合：

- 用户明确知道 locator 可用。
- 页面状态导致 Runner 暂时无法验证。
- 先保存后再用元素库批量验证补证据。
