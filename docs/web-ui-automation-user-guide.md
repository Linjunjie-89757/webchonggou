# Web UI 自动化试用说明

这份文档给试用同事使用，目标是帮助大家完成第一条 Web UI 用例的创建、调试、运行和失败排查。当前版本已经支持手工维护步骤、调试运行、正式运行、查看步骤日志、失败截图和报告证据；AI 生成步骤入口已预留，但本阶段暂不启用。

## 1. 环境配置

### 1.1 启动前后端

1. 启动后端服务，确保接口地址可访问：`http://localhost:8080`。
2. 启动前端服务，访问前端地址，例如：`http://localhost:5176`。
3. 登录系统后进入对应工作空间，再进入“Web UI 自动化”页面。

### 1.2 浏览器执行环境

Web UI 自动化执行引擎基于 Playwright。试用前确认：

- 后端机器已安装 Playwright 运行所需浏览器。
- 后端进程有权限访问被测系统地址。
- 如果用例需要访问内网系统，后端机器也必须能访问该内网地址。
- 如果执行失败提示浏览器缺失，需要在后端环境安装 Playwright 浏览器依赖。

### 1.3 基础地址

用例里可以填写“起始地址”：

- 如果步骤 `OPEN` 写完整 URL，例如 `https://example.com/login`，会直接打开该地址。
- 如果步骤 `OPEN` 写相对路径，例如 `/login`，执行时会和用例或环境里的基础地址组合。
- 建议同一个系统的用例统一配置基础地址，步骤里只写相对路径，后续切换环境更方便。

## 2. 创建用例

1. 进入“Web UI 自动化”页面。
2. 点击“新建用例”。
3. 填写用例名称、模块、起始地址、浏览器、默认超时时间。
4. 在步骤表格里点击“新增步骤”。
5. 依次维护步骤名称、类型、定位方式、定位值、输入/目标值、截图策略。
6. 点击“保存”。

保存前系统会校验：

- 用例名称不能为空。
- 至少要有 1 个步骤。
- 不能所有步骤都禁用。
- `OPEN` 必须填写打开地址。
- 需要定位器的步骤必须填写定位方式和定位值。
- `FILL` 必须填写输入文本。
- `ASSERT_TEXT` 必须填写期望文本。

## 3. 步骤类型说明

| 步骤类型 | 用途 | 必填项 |
| --- | --- | --- |
| 打开页面 `OPEN` | 打开完整 URL 或相对路径 | 输入/目标 |
| 点击 `CLICK` | 点击按钮、链接、菜单等元素 | 定位方式、定位值 |
| 输入 `FILL` | 给输入框填写文本 | 定位方式、定位值、输入文本 |
| 清空 `CLEAR` | 清空输入框或可编辑元素 | 定位方式、定位值 |
| 等待 `WAIT_FOR` | 等待元素出现 | 定位方式、定位值 |
| 可见断言 `ASSERT_VISIBLE` | 验证元素是否可见 | 定位方式、定位值 |
| 文本断言 `ASSERT_TEXT` | 验证元素包含期望文本 | 定位方式、定位值、期望文本 |
| 截图 `SCREENSHOT` | 保存当前页面截图 | 无 |

## 4. 定位方式建议

优先级建议如下：

1. `测试 ID`：最稳定，建议前端给关键按钮、输入框增加 `data-testid`。
2. `Label` / `Placeholder`：适合表单输入框，语义清楚。
3. `角色`：适合按钮、链接等可访问性语义明确的元素。
4. `文本`：适合页面上唯一且稳定的中文文案。
5. `CSS`：适合结构稳定的页面，建议避免过长层级选择器。
6. `XPath`：只在复杂 DOM 或没有更好定位方式时使用。

不建议使用容易变化的定位：

- 自动生成的 class。
- 过长的 DOM 层级，例如 `div > div > div:nth-child(3)`。
- 依赖列表顺序的定位，除非业务上顺序固定。
- 会随语言、配置或数据变化的文本。

## 5. 调试运行

编辑器里支持两种调试方式：

- “调试运行”：按当前用例全部步骤执行。
- 单步调试：只执行当前步骤，适合快速验证定位器或输入值。

如果当前编辑内容有未保存修改，系统会提示“本次调试会按当前草稿内容执行，但不会自动保存到用例”。这代表：

- 调试会使用你当前页面上看到的内容。
- 调试成功不等于已经保存。
- 确认用例可用后，还需要点击“保存”。

调试完成后会自动打开执行报告抽屉。失败时可以查看：

- 失败步骤。
- 步骤类型。
- 定位器。
- 输入/目标值。
- 错误信息。
- 截图证据。

## 6. 正式运行和报告截图

在用例列表点击“运行”，系统会按保存后的用例内容执行，并打开报告。

报告里重点看三块：

- 顶部概要：用例、浏览器、耗时、成功/失败/跳过步骤数。
- 失败证据：失败步骤会集中展示，便于快速排查。
- 步骤明细表：每一步都有状态、定位器、输入值、耗时、错误信息和截图入口。

如果失败步骤有截图，点击“打开失败截图”或表格里的“截图”即可查看证据。

报告里的“定位步骤”用于快速回到编辑器：

- 已保存用例的报告会打开对应用例编辑器，并提示失败步骤序号。
- 草稿调试没有保存用例 ID 时，会提示回到当前编辑器查看调试结果。

## 7. 常见失败排查

### 7.1 打不开页面

可能原因：

- 基础地址为空，`OPEN` 又写了相对路径。
- 后端机器无法访问被测系统。
- 被测系统需要 VPN、内网或特殊 hosts。
- URL 写错或环境配置错误。

处理建议：

- 先用完整 URL 调试 `OPEN` 步骤。
- 在后端机器上确认能访问该地址。
- 确认环境配置里的基础地址是否正确。

### 7.2 找不到元素

可能原因：

- 定位器写错。
- 页面还没加载完成。
- 元素在 iframe、弹窗或异步区域里。
- 文案或 DOM 结构发生变化。

处理建议：

- 优先改用 `测试 ID`、`Label`、`Placeholder` 或 `角色`。
- 增加 `WAIT_FOR` 步骤等待关键元素出现。
- 避免使用过长 CSS 或 XPath。
- 查看失败截图确认页面实际停在哪一步。

### 7.3 点击无效

可能原因：

- 元素被遮挡。
- 点到了相似元素。
- 页面弹窗、loading 或权限提示阻塞。
- 定位器匹配到多个元素中的第一个，但不是目标元素。

处理建议：

- 用更精确的定位器。
- 在点击前增加 `ASSERT_VISIBLE` 或 `WAIT_FOR`。
- 查看截图确认点击前页面状态。

### 7.4 输入失败

可能原因：

- 定位到的不是输入框。
- 输入框只读或被禁用。
- 输入框需要先点击、清空或等待。

处理建议：

- 用 `Label`、`Placeholder` 或 `测试 ID` 定位输入框。
- 输入前增加 `CLEAR`。
- 必要时先 `CLICK` 输入框再 `FILL`。

### 7.5 文本断言失败

可能原因：

- 期望文本写错。
- 页面返回了错误提示或空数据。
- 文本出现较慢。
- 定位到了父容器或错误元素。

处理建议：

- 先看截图确认页面实际文本。
- 使用更稳定、更小范围的定位器。
- 在断言前增加 `WAIT_FOR`。
- 期望文本尽量填写关键短语，不要填整段易变化文案。

## 8. 试用反馈建议

试用时建议记录这些信息：

- 哪个用例、哪一步失败。
- 失败报告里的错误信息。
- 截图是否足够判断问题。
- 定位方式是否好理解。
- 哪些步骤类型还不够用。
- 是否需要批量运行、定时运行或接入流水线。

这些反馈会用于下一阶段决定是否接入 AI 生成步骤、批量执行和更完整的执行趋势报表。
